# AgentOps 小助手：飞书日志分析与故障修复审批

> 状态: 待扩展

## 模块职责

在飞书中触发日志分析与故障修复审批，实现双级漏斗意图防御网格。

---

## 待实现

### 问题

在当前的 AgentOps 实现中，飞书机器人的每次对话都会通过 `go b.handleAgentRun(chatId, prompt)` 开启一个新的后台 Goroutine 去跑 Main Loop。

在实际的团队运维群中，大家可能会聊很多与运维无关的天（比如："今天中午吃什么？"或者只是群员之间的互相吐槽）。如果机器人对群里的每一句话都触发一次昂贵的大模型 Main Loop 进行回应，不仅极大地浪费 API Token，还会严重干扰 Agent 正在进行的真正排障任务。

**结合我们在驾驭工程中学到的知识**，如果要为 go-tiny-claw 增加一个"意图拦截过滤器（Intent Filter）"，只有当用户的话语中明确包含需要 Agent 介入的意图（比如包含 @机器人 或者明确要求执行物理操作）时，才唤醒 Main Loop；否则只是简单忽略。你会选择将这个拦截器做在哪个架构层（飞书 Dispatcher 接收层、还是作为单独调一个小模型的前置网关）？为什么？

### 方案：双级漏斗意图防御网格

我们将拦截器死死钉在**飞书 Dispatcher 接收层**（即主引擎 Main Loop 启动之前）。流量必须通过两道关卡的物理审计，通不过的直接在接收层静默丢弃，绝对不往下游投递。

#### 第一级：接收层硬编码规则（零成本，瞬时物理蒸发）

在飞书 Webhook 回调函数（`OnP2MessageReceiveV1`）接入的第一时间，启动纯代码级别的强规则静态匹配。这一步追求的是绝对的快与省。

**过滤三板斧**：

- **门禁 A（@ 机器人强校验）**：检查飞书事件中的消息正文或提及列表（Mention List）。如果消息是在大群里发的，且没有显示 @Agent机器人，说明是人类之间的日常扯淡，拦截器直接 `return nil`，物理蒸发，不留任何痕迹。

- **门禁 B（显式操作指令前缀）**：维护一个高频运维动作的正则/关键词硬编码池（如 `^排查、^查日志、^修复、^status`）。如果用户说的话命中了这些硬前缀，说明意图极其明确，直接开绿灯绕过第二级小模型，瞬间唤醒主 Main Loop。

- **门禁 C（黑名单直杀）**：纯闲聊的高频词（如"哈哈"、"收到"、"谢谢"、"吃什么"），直接判定为无效噪音，就地拦截。

#### 第二级：接收层外挂开源超轻量小模型（低成本，模糊意图嗅探）

当一条消息满足了"艾特了机器人"，但说话内容比较模糊（例如："@智能助手 感觉线上有点卡，你看看怎么回事" 或 "@智能助手 辛苦了，你真棒"），第一级硬编码无法做出复杂的语义断言。

此时，在接收层内部分流，同步调用一个企业私有化部署的、极轻量级的开源小模型（如 Qwen-1.5B / 7B 或 GLM-4-Flash）：

**极简判定 Prompt 模板设计**：

```
Role: 你是一个高密度的"ChatOps 意图合规审查网格"。

Task: 判断用户这句话是否包含"需要运维机器人对服务器进行实际排查、读取、命令执行或故障分析"的工作意图。

Output Format: 只能输出 "TRUE" 或 "FALSE"，严禁输出任何多余字符。
- 如果是闲聊、赞美、无意义的寒暄（如"你真棒"、"你好"），输出 "FALSE"。
- 如果要求看日志、查状态、重构、修Bug（如"看看怎么卡了"），输出 "TRUE"。

User Input: "{{USER_PROMPT}}"
```

**拦截流转闭包**：

- **FALSE**：小模型认为在闲聊。接收层拦截，并调用飞书 API 极其礼貌且调皮地回复一句："收到！不过目前系统看起来很健康哦，等您下达具体的排障或指令时我再全力出击！🫡"。主 Main Loop 依然处于冬眠状态，Session 历史纯净如初。

- **TRUE**：小模型确认这是真实的生产需求。接收层立刻释放闸门，将原始 Prompt 投递给下游，正式唤醒高智商的主大模型进入 ReAct 慢思考。
