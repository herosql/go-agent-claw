# 错误自愈：上下文感知的 Error Recovery 提示模板注入

> 状态: 待扩展

---

## 背景与问题

在云原生运维场景中，Agent 遇到复杂报错（如 CrashLoopBackOff 长篇堆栈）时可能束手无策。如何用"AI 治愈 AI"？

---

## 核心方案：双轨制错误自愈架构

### 架构拓扑

```
工具报错 → Error 数据 → RecoveryEngine → 分流
                                      ├── 第一轨：Domain Error Code 匹配
                                      └── 第二轨：小模型 Summary 兜底
```

### 第一轨：Domain Error Code（确定性防线）

**规范化错误定义**：

| 领域错误码 | 触发场景 | 注入的"行动指南"模板 |
|-----------|----------|---------------------|
| ERRFILENOTFOUND | readfile 报 OS 级别不存在 | "目标路径不存在。请勿盲目猜测，必须先执行 bash(ls -la) 确认真实目录结构。" |
| ERREDITMATCHFAILED | editfile 找不到 oldtext | "原文件内容已变更。请先重新调用 readfile 刷新工作区内存，严禁盲目重试。" |
| ERRBASHTIMEOUT | 物理命令执行超过 30s 强杀 | "进程被超时强杀。若为常驻服务，请重新调用 bash 并将其转入后台（如 nohup ... &）。" |

**运行机制**：
- 极致性能：纯指针或字符串映射，O(1) 微秒级，0 Token 消耗
- 强力锚定：祈使句主导，强制校准下一步 Tool 调用走向

### 第二轨：私有化小模型 Summary（长尾未知防线）

当第一轨返回 ERR_UNKNOWN 时激活（如 K8s 闪退、Coredump、复杂编译网络报错）。

**专属 Prompt 模板**：

```
Role: 你是一个高密度的"编译与运维报错提炼专家"。
Task: 你将看到一段复杂的系统底层错误日志。请用一句话（不超过50字）精准提炼出：
      1. 核心原因是什么
      2. 建议执行什么具体动作解决
Constraints:
  1. 严禁展开长篇大论，严禁解释原理
  2. 必须直接给出具备可操作性的工具调用建议
  3. 如果你完全看不懂，直接输出 "UNKNOWN"
```

**流控与拦截**：
- 内网私有化部署，零 API 计费，数据不出机房
- 硬性超时 1.5 秒，超时强杀，原样抛出原始错误

**弱化标签注入**：
```
[Harness 诊断层提示(仅供参考)]: 检测到长尾异常，系统推测原因为【小模型的 Summary】。请结合工作区物理状态核对后做出决策。
```

### 工业级工程防御：防止自愈死循环

| 防护机制 | 规则 |
|----------|------|
| 同一错误限制 | 单次会话中最多允许注入 2 次自愈提示 |
| 强制阻断（Break-out） | 阈值 3 时彻底熔断，触发人工审批信号 |
